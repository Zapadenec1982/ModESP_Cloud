# ModESP Cloud — Можливості платформи

**Мультитенантна IoT-платформа для моніторингу комерційного холодильного обладнання, відповідності HACCP та дистанційного керування пристроями.**

> Побудовано на Node.js, PostgreSQL, MQTT, Svelte — розгорнуто на Linux VPS із TLS на всіх рівнях.

---

## Загальний огляд

| Показник | Значення |
|----------|----------|
| API-ендпоінти | 60+ REST |
| Канали реального часу | MQTT + WebSocket |
| Параметри пристрою | 48 на пристрій (температури, аварії, налаштування) |
| Типи аварій | 10 (критичні / попередження / інформаційні) |
| Тестове покриття | 130+ інтеграційних тестів |
| Мови інтерфейсу | Українська, Англійська |
| Розгортання | Продакшн на Hetzner VPS, TLS/HTTPS |

## 1. Мультитенантність

Повна ізоляція орендарів на кожному рівні — база даних, MQTT-брокер, API та інтерфейс.

- **CRUD орендарів** — створення, перейменування, деактивація орендарів з тарифними планами (free / basic / pro / enterprise)
- **Ізоляція даних** — кожен SQL-запит обмежений `tenant_id`; MQTT-топіки фізично розділені за slug орендаря
- **Крос-тенантні операції** — суперадмін бачить усіх орендарів, переміщує пристрої між ними
- **RBAC з урахуванням орендаря** — користувачі належать до одного або кількох орендарів; адмін бачить лише дані свого орендаря
- **Перемикання орендарів** — мультитенантні користувачі обирають активного орендаря при вході або перемикаються під час сесії

## 2. Управління пристроями

Повний життєвий цикл від виробництва до експлуатації — автовиявлення, призначення, моніторинг, перепризначення.

### Автовиявлення та провізіонінг
- ESP32-пристрої самореєструються через MQTT із спільним bootstrap-ключем
- Нові пристрої з'являються в черзі **Очікуючих пристроїв** зі статусом онлайн/офлайн
- Адмін призначає пристрій орендарю → платформа автоматично генерує унікальні MQTT-облікові дані та доставляє їх по повітрю (OTA)
- Провізіонінг без втручання: пристрій перепідключається з новими обліковими даними автоматично

### Властивості пристрою
- Назва, розташування, серійний номер, модель, дата виробництва, коментарі
- Версія прошивки та версія протоколу відстежуються через heartbeat
- Статус онлайн/офлайн у реальному часі (поріг heartbeat — 90 секунд)

### Стан у реальному часі
- 48 параметрів оновлюються в реальному часі: температури (повітря, випарника, конденсатора), уставка, стан компресора, цикл розморожування, стан дверей, захисні аварії, налаштування термостата
- Стан відображається в UI з групуванням за категоріями (Обладнання, Термостат, Розморожування, Захист)
- WebSocket push — інтерфейс оновлюється миттєво без опитування

### Перепризначення пристрою
- Суперадмін переміщує пристрій між орендарями в один клік
- Автоматична ротація облікових даних, очищення RBAC та міграція MQTT-топіків
- 120-секундний grace-період запобігає хибним сповіщенням про офлайн під час переходу

## 3. Дистанційне керування

Надсилання команд на пристрої з хмари — через REST API або веб-інтерфейс.

- **Редагування параметрів** — зміна уставки термостата, інтервалів розморожування, порогів захисту дистанційно
- **Валідовані команди** — приймаються лише параметри, дозволені для запису (визначені в схемі метаданих пристрою)
- **Повне оновлення стану** — запит на повторну публікацію всіх 48 параметрів на вимогу
- **MQTT-доставка** — команди публікуються в MQTT-топіки конкретного пристрою з гарантіями QoS
- **Рольовий доступ** — лише ролі admin та technician можуть надсилати команди; viewers — тільки перегляд

## 4. Телеметрія та аналітика

Серверне семплювання з гнучкими запитами та агрегацією часових рядів.

### Збір даних
- 6 каналів телеметрії з семплюванням кожні 5 хвилин: температура повітря, температура випарника, температура конденсатора, уставка, компресор (вкл/викл), розморожування (вкл/викл)
- 30-секундний дебаунсинг запобігає дублюванню записів при швидких змінах стану
- Щомісячні партиції PostgreSQL — автоматичне створення та очищення через 90 днів

### API запитів
- **Сирі дані** — до 10 000 точок за запит із заголовком `X-Truncated` при обрізанні
- **Агреговані статистики** — min / max / avg за часовий бакет (5хв, 15хв, 1год, 6год, 1день)
- **Фільтрація каналів** — запит лише потрібних каналів
- **Гнучкий часовий діапазон** — ISO-мітки часу або відносні години (за замовчуванням 24 год, максимум 31 день)

### Інтерактивні графіки
- Багатоканальний лінійний графік (uPlot) із зумом, паном, легендою
- Накладення подій — цикли компресора, старти розморожування, аварії відображаються як пунктирні вертикальні лінії
- Розгортуваний журнал подій під графіком

## 5. Відповідність HACCP та експорт даних

Вбудовані інструменти для відповідності вимогам харчової безпеки (регуляції HACCP України).

### Експорт CSV
- **CSV телеметрії** — журнали температур із мітками часу, UTF-8 BOM для сумісності з Excel
- **CSV інвентаризації пристроїв** — усі пристрої з властивостями (назва, модель, розташування, серійний номер, прошивка)
- **CSV історії аварій** — з фільтрацією за серйозністю та діапазоном дат (до 90 днів)

### PDF-звіт
- **Температурний звіт HACCP** — професійний PDF із зведеною статистикою, хронологією аварій, погодинним журналом температур
- Підтримка кирилиці (шрифт Roboto) — готовий для подання в українські регуляторні органи
- Генерація на сервері (pdfmake) — без залежності від браузера

### Обмеження запитів
- 10 експортів на хвилину на користувача — запобігає зловживанню без блокування легітимного використання

## 6. Система аварій

10 типів аварій із класифікацією за серйозністю, затримками для уникнення хибних спрацювань та багатоканальними сповіщеннями.

### Типи аварій та серйозність

| Серйозність | Аварії |
|-------------|--------|
| **Критична** | Висока температура, Низька температура, Відмова датчика 1, Відмова датчика 2 |
| **Попередження** | Відкриті двері, Безперервна робота, Збій охолодження |
| **Інформаційна** | Аварія швидкості зміни, Короткий цикл, Швидкий цикл |

### Розумне виявлення
- Моніторинг переходів булевих станів (off→on = аварія піднята, on→off = знята)
- **Затримки проти хибних спрацювань** — аварія дверей чекає 2 хвилини, аварія охолодження — 5 хвилин перед підтвердженням
- Мітки часу активації/зняття з відстеженням тривалості

### API аварій та UI
- Фільтрація за серйозністю, активні/зняті, пристрій, діапазон дат
- Статистика аварій — кількість та середня тривалість за кодом аварії
- Фільтри серйозності в UI (Усі / Критичні / Попередження / Інформаційні) для швидкого аналізу
- RBAC на рівні пристрою — користувачі бачать аварії лише для призначених пристроїв

## 7. Енергомоніторинг

Розрахункове енергоспоживання на основі профілів потужності моделей обладнання.

### Оцінка енергоспоживання
- Час роботи компресора x номінальна потужність → розрахункові кВт·год за інтервал семплювання
- Розбивка за компонентами: компресор, нагрівач відтайки, вентилятори, споживання в режимі очікування
- Профілі моделей обладнання (таблиця device_models) з індивідуальними перевизначеннями потужності на пристрій
- Розрахунок вартості з налаштовуваним тарифом на електроенергію для кожної організації

### Сумісність з майбутніми датчиками
- Зарезервований MQTT-ключ `equipment.energy_kwh` для реальних CT clamp датчиків енергії (майбутня прошивка)
- Автовиявлення: якщо прошивка публікує `equipment.energy_kwh`, семплер використовує реальне значення замість розрахункового
- Прапорець `energy_source`: `estimated` (за замовчуванням) або `metered` (CT clamp)

### UI та API
- Вкладка Енергія на сторінці деталей пристрою — графік кВт·год, зведення вартості, розбивка за компонентами
- Канал енергії (`energy`) на графіку телеметрії поряд з каналами температур
- `GET /api/devices/:id/energy/summary` — підсумки кВт·год, вартість, розбивка за компонентами

## 8. Відстеження подій

Операційні події за межами аварій — цикли обладнання, зміни статусу, з'єднання пристроїв.

- **Типи подій** — компресор вкл/викл, розморожування старт/кінець, аварія піднята/знята, пристрій онлайн/офлайн
- **API запитів** — фільтрація за типом події, часовим діапазоном, пагінація
- **Накладення на графік** — події відображаються як вертикальні маркери на графіках телеметрії
- **Буферизований запис** — інтервал скидання 1 секунда з масовим INSERT для продуктивності

## 9. Сповіщення та алертинг

Багатоканальна система push-сповіщень — Telegram, Firebase (мобільний), Web Push.

### Telegram-бот
- Повнофункціональний бот із 7 командами: `/start`, `/devices`, `/status`, `/alarms`, `/tenant`, `/unlink`, `/help`
- **Прив'язка користувача** — згенерувати 7-символьний код у веб-інтерфейсі → надіслати боту → обліковий запис прив'язаний
- **Статус пристрою** — натиснути на пристрій → детальний перегляд з температурами, розташуванням, аваріями
- **Мультитенантність** — перемикання активного орендаря через команду `/tenant`
- **Двомовність** — українська та англійська з перемиканням мови для кожного чату
- **З урахуванням RBAC** — техніки бачать лише призначені пристрої; інформаційні аварії → тільки для адміна

### Firebase Cloud Messaging
- Push-сповіщення для Android та вебу
- Автоматичне очищення застарілих FCM-токенів

### Маршрутизація сповіщень
- Фільтр пристроїв для кожного підписника — вибір, які пристрої запускають сповіщення
- Перемикач активний/неактивний — м'яке вимкнення без видалення
- **Дебаунсинг** — 5-секундний кулдаун на пристрій+аварію для запобігання дублікатам
- **Затримка офлайну** — 2-хвилинне очікування перед відправкою "пристрій офлайн" (запобігає шуму від нестабільного з'єднання)

### Відстеження доставки
- Журнал сповіщень зі статусом, повідомленнями про помилки, мітками часу
- Тестовий ендпоінт для перевірки з'єднання підписника

## 10. OTA-оновлення прошивки

Завантаження, розгортання та моніторинг оновлень прошивки — один пристрій або масове розгортання на флот.

### Бібліотека прошивок
- Завантаження `.bin` файлів (до 4 МБ) з тегом версії, типом плати, примітками до релізу
- SHA-256 контрольна сума обчислюється при завантаженні, перевіряється пристроєм при скачуванні
- Перевірка сумісності плати — запобігає розгортанню неправильної прошивки на неправильне обладнання

### Розгортання на один пристрій
- Розгортання в один клік з UI або API
- MQTT-команда з URL завантаження, версією, контрольною сумою
- Відстеження статусу: в черзі → відправлено → успіх / невдача (тайм-аут 10 хвилин)
- Версія до OTA зберігається для надійного виявлення успіху через heartbeat

### Групове розгортання
- Обрати прошивку + список пристроїв → розгортання в налаштовуваних пакетах
- **Розмір пакету** — скільки пристроїв на хвилю
- **Інтервал пакетів** — секунди між хвилями (запобігає перевантаженню мережі)
- **Поріг невдач** — автоматична пауза розгортання, якщо відсоток невдач перевищує налаштоване значення
- Адмін може відновити призупинене розгортання
- Витримує перезапуск сервера — відновлюється з бази даних при старті

### Фоновий моніторинг
- Опитування кожні 30 секунд: порівняння версії прошивки пристрою з очікуваною після OTA
- Автоматичне виявлення успіху/невдачі без зворотного виклику від пристрою

## 11. Управління користувачами та автентифікація

Автентифікація на основі JWT із 4-рівневим RBAC та контролем доступу на рівні пристрою.

### Автентифікація
- **JWT-токени** — access-токен на 15 хвилин, refresh-токен на 30 днів із ротацією
- **Політика паролів** — мінімум 15 символів (відповідно до NIST SP 800-63B, без вимог до складності)
- **Перевірка HaveIBeenPwned** — клієнтська k-anonymity перевірка проти бази скомпрометованих паролів
- **Обмеження запитів** — 50 спроб входу / 5 хвилин / IP
- **Зміна пароля** — вимагає верифікацію старого пароля, видає нові токени

### Рольовий контроль доступу (RBAC)

| Роль | Область | Можливості |
|------|---------|------------|
| **Суперадмін** | Уся платформа | Усі операції, крос-тенантний доступ, аудит-лог, CRUD орендарів |
| **Адмін** | Свій орендар | Повний контроль: пристрої, користувачі, прошивки, сповіщення |
| **Технік** | Призначені пристрої | Перегляд, надсилання команд, розгортання прошивки, сервісні записи |
| **Спостерігач** | Призначені пристрої | Доступ лише для читання (без команд, без редагування) |

### Призначення пристроїв
- Адмін призначає конкретні пристрої технікам/спостерігачам
- Контроль на рівні API — несанкціонований доступ до пристрою повертає 403
- Масове призначення/відкликання через API
- Аудит-слід призначень (ким надано, коли надано)

### Мультитенантне членство
- Користувачі можуть належати до кількох орендарів (зв'язок M:N)
- Вибір орендаря при вході, якщо є кілька членств
- Перемикання орендаря під час сесії без повторного входу

### Прив'язка Telegram
- Згенерувати код прив'язки у веб-інтерфейсі → надіслати Telegram-боту → обліковий запис з'єднаний
- Команди бота враховують RBAC-роль та призначення пристроїв користувача

## 12. Аудит-логування

Незмінний, append-only журнал аудиту для відповідності та безпеки.

- **Автоматичне захоплення** — усі операції створення/оновлення/видалення логуються без змін коду в маршрутах
- **Записані поля** — користувач, дія, тип/ID сутності, зміни до/після (JSON diff), IP-адреса, user agent, мітка часу
- **Незмінність** — тригер PostgreSQL забороняє UPDATE/DELETE в таблиці аудит-логу
- **API запитів** — фільтрація за орендарем, типом сутності, дією, користувачем, діапазоном дат (тільки суперадмін)
- **Веб-інтерфейс** — сортована таблиця з фільтрами, переглядач JSON diff для змін до/після
- **Fire-and-forget** — записи аудиту ніколи не блокують основний запит

## 13. Комунікація в реальному часі

Два канали реального часу — MQTT для пристрій-хмара, WebSocket для хмара-браузер.

### MQTT (Пристрій ↔ Хмара)
- Ієрархія топіків: `modesp/v1/{орендар}/{пристрій}/state/{ключ}` (48 індивідуальних скалярних топіків)
- Heartbeat кожні 30 секунд із версією прошивки, аптаймом, вільним heap, RSSI WiFi
- TLS-шифрування на порту 8883 із сертифікатом Let's Encrypt
- Брокер Mosquitto з PostgreSQL-backed ACL (плагін go-auth)

### WebSocket (Хмара → Браузер)
- JWT-автентифіковане з'єднання (`ws://host/ws?token=...`)
- Обмеження за орендарем — клієнти отримують оновлення лише для пристроїв свого орендаря
- Типи повідомлень: дельта стану (лише змінені ключі), аварія, статус пристрою
- RBAC на рівні пристрою — підписка перевіряється відповідно до призначень пристроїв користувача
- Обробка протитиску — пропуск повідомлень, якщо буфер клієнта перевищує 64 КБ

## 15. Інфраструктура та експлуатація

Готове до продакшну розгортання з TLS, резервними копіями та моніторингом.

### Безпека
- HTTPS скрізь (Let's Encrypt, автоматичне поновлення)
- MQTT TLS на порту 8883
- ESP32 перевіряє серверний сертифікат через вбудований набір CA
- Хешування паролів bcrypt (фактор вартості 12)
- CORS обмежений продакшн-доменом
- Незмінний аудит-лог

### База даних
- PostgreSQL 16 з пулом з'єднань (макс. 30 з'єднань)
- Тайм-аут запитів (30с) запобігає зависанню запитів
- Щомісячні партиції телеметрії з автоматичним створенням та очищенням через 90 днів
- 18+ таблиць з індексами, зовнішніми ключами та обмеженнями

### Моніторинг
- `GET /api/health` — статус бази даних, MQTT, аптайм
- Моніторинг StateMap — кількість пристроїв, загальна кількість ключів, оцінка використання пам'яті (логується кожні 60с)
- Структуроване логування Pino (JSON у продакшні)

### Резервне копіювання та обслуговування
- Щоденний дамп PostgreSQL о 2:00 (зберігання 30 днів)
- Очищення партицій телеметрії о 3:00 щодня
- Попереднє створення місячних партицій (25-го числа кожного місяця)

### Розгортання
- systemd-сервіс з автоматичним перезапуском
- Nginx reverse proxy з підтримкою WebSocket upgrade
- Розгортання через Git (`git pull` + `systemctl restart`)
