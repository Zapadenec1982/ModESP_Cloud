# ModESP Cloud — Дорожня карта

## Поточний стан

**Розгорнуто у продакшні ✅ — 13 фаз завершено, ESP32 підключений через MQTT+TLS**

Завершено: хмарна основа, REST API, WebSocket, веб-інтерфейс, push-сповіщення (FCM+Telegram+WebPush), автентифікація (JWT), історія та аналітика, OTA-оновлення флоту, інтернаціоналізація (UA/EN), рольовий доступ до пристроїв, масштабування, динамічна MQTT-автентифікація (go-auth), керування орендарями, мультитенантні користувачі, редизайн Telegram-бота, аудит-логування, тестова інфраструктура (130+ тестів), зміцнення платформи (Events API, HACCP експорт, зміна пароля, класифікація аварій), енергомоніторинг.

**Далі: Фаза 12 — Масовий імпорт пристроїв**

## Завершені фази

### Фаза 1: Хмарна основа ✅
Базова інфраструктура — ESP32 підключається до хмари, дані зберігаються.

- Зміни прошивки (ModESP_v4): поле орендаря в NVS, генератор префіксу, heartbeat, обробник `_set_tenant`
- VPS (Ubuntu 24), Mosquitto (ACL + TLS), схема PostgreSQL
- MQTT-сервіс: парсинг топіків, агрегація стану (48 ключів), детектор аварій, семплер телеметрії (5 хв), детектор подій
- Nginx HTTPS, systemd-сервіси, резервне копіювання, партиціонування телеметрії

### Фаза 2: Віддалений моніторинг ✅
Технік бачить стан усіх контролерів з будь-якої точки світу.

- WebSocket (трансляція дельт у реальному часі по орендарю)
- REST API (пристрої, телеметрія, аварії, команди)
- Svelte WebUI: Дашборд, Деталі пристрою, Очікуючі пристрої
- Інтерфейс автовиявлення (очікуючий → призначення)

### Фаза 3: Push-сповіщення ✅
Технік отримує сповіщення про аварії миттєво.

- Оркестратор сповіщень з дебаунсингом та реєстром каналів
- FCM + Telegram-бот + Web Push (VAPID)
- REST API: CRUD підписників, тестова відправка, журнал доставки
- Веб-інтерфейс: сторінка сповіщень

### Фаза 4: Автентифікація та управління користувачами ✅
Кілька техніків, кілька організацій, контроль доступу.

- JWT (вхід, ротація refresh-токенів, вихід), 4 ролі (суперадмін/адмін/технік/спостерігач)
- CRUD користувачів, JWT-автентифікація WebSocket, сторінки входу та користувачів
- mosquitto-go-auth з PostgreSQL ACL (замість статичного ACL)
- MQTT Bootstrap Provisioning: спільний ключ → унікальні облікові дані при призначенні
- Автоматичне виявлення "застряглих" пристроїв (120с grace → автоскидання)
- Життєвий цикл пристрою: м'яке скидання (active→pending) + повне видалення

### Фаза 5: Історія та аналітика ✅
Аналіз трендів, виявлення деградації обладнання.

- Статистика телеметрії: агрегація по бакетах (5хв/15хв/1год/6год/1день), статистика аварій, зведення флоту
- Графіки uPlot, таблиця історії аварій, панель зведення флоту
- Партиціонування PostgreSQL по місяцях, зберігання 90 днів

### Фаза 6: OTA-оновлення флоту ✅
Оновлення прошивки на всіх пристроях без виїзду на об'єкт.

- Завантаження прошивки (контрольна сума SHA256), розгортання на один пристрій + групове з пакетуванням
- Автоматична пауза при перевищенні порогу невдач, перевірка сумісності плати
- OTA-обробник ModESP_v4: HTTP завантаження → SHA256 → запис → перезавантаження (~8с)
- Веб-інтерфейс: сторінка прошивок (завантаження, розгортання, моніторинг)

### Фаза 6.5: Шліфування інтерфейсу ✅
Зручний інтерфейс для техніків.

- Інтернаціоналізація (UA + EN), світла/темна тема
- Метадані пристрою (модель, коментар, дата виробництва), сервісні записи
- Модалка редагування, пошук за всіма полями

### Фаза 7: Рольовий доступ + масштабування ✅
Масштабування до 5000+ пристроїв, контроль доступу на рівні пристрою.

- **7a:** Рольовий доступ — middleware фільтрації та перевірки доступу, перевірка в WebSocket
- **7b:** Масштабування — пул БД (30), пакетний запис стану, дедуплікація heartbeat, пакетування подій, протитиск WebSocket (64КБ)
- **7c:** Фронтенд RBAC — сховища isAdmin/canWrite, умовний UI, захист маршрутів, модалка призначення пристроїв
- **7d:** Сумісність OTA — тип плати прошивки, валідація при розгортанні, фільтрація при груповому розгортанні

### Фаза 8a: Управління орендарями ✅
Роль суперадміна, крос-тенантні операції.

- CRUD API орендарів, перепризначення пристрою (ротація MQTT-облікових даних + _set_tenant через старий slug)
- Сторінка орендарів, модалка "Змінити орендаря" у деталях пристрою

### Фаза 8b: Мультитенантні користувачі ✅
Один користувач належить кільком орендарям (M:N).

- Таблиця зв'язків user_tenants, потік pendingToken
- Вхід → вибір орендаря → перемикання орендаря
- Веб-інтерфейс: перемикач орендарів у бічній панелі, модалка управління орендарями

### Фаза 8c: Редизайн Telegram-бота ✅
Повнофункціональний Telegram-бот з автентифікацією і RBAC.

- Автентифікація через код прив'язки, 7 команд, рольовий доступ до пристроїв
- Підтримка мультитенантності (перемикання через /tenant)
- Сповіщення: аварія виникла + знята + пристрій офлайн — з розташуванням
- Постійна клавіатура, двомовність UA/EN, автоочищення чату

### Фаза 9: Аудит-логування ✅
Аудит усіх мутацій для відповідності та безпеки.

- Таблиця audit_log (тригер незмінності, 4 індекси)
- Middleware: автозахоплення POST/PUT/PATCH/DELETE (fire-and-forget)
- 15 точок збагачення (req.auditContext зі змінами до/після)
- Веб-інтерфейс: сторінка аудит-логу (фільтри, пагінація, JSON diff)

### Фаза 10: Тестова інфраструктура ✅
130+ інтеграційних тестів на реальній PostgreSQL.

- Vitest 3.2 + Supertest + Docker Compose (PostgreSQL 5433, tmpfs)
- 15 тестових наборів: автентифікація, RBAC, ізоляція орендарів, CRUD, аудит, OTA, сповіщення
- Допоміжні модулі: app.js, factories.js, запускач міграцій

### Фаза 11: Зміцнення платформи ✅
Закриття розривів виявлених аудитом — HACCP, NIST, ISA-18.2.

- **11a:** Events API + накладення подій на графік (компресор/розморожування/аварії)
- **11b:** HACCP експорт даних — CSV (телеметрія, пристрої, аварії) + PDF-звіт (кирилиця)
- **11c:** Інтерфейс зміни пароля + політика NIST (мін. 15 символів, перевірка HaveIBeenPwned)
- **11d:** Класифікація аварій за серйозністю (критична/попередження/інформаційна, затримки, фільтр)

### Фаза 13: Енергомоніторинг ✅ (2026-03-24)
Оцінка споживання енергії на основі енергопрофілів моделей обладнання.

- Таблиця device_models з профілями потужності (компресор/відтайка/вентилятори/standby у ватах)
- Семплер телеметрії розраховує оцінені кВт·год з стану компресора/відтайки/вентиляторів × номінальна потужність
- Канал телеметрії `energy` зберігається поряд з каналами температури
- Індивідуальні override потужності (devices.power_overrides JSONB)
- Калькуляція вартості з конфігурованим тарифом на електроенергію по організації
- API зведення енергії: `GET /devices/:id/energy/summary` (кВт·год, вартість, розбивка)
- CRUD API моделей обладнання: `GET/POST/PATCH/DELETE /device-models`
- Вкладка Енергія на сторінці пристрою + канал енергії на графіку телеметрії
- Сумісність з майбутнім: зарезервований MQTT-ключ `equipment.energy_kwh` для датчиків струму (автовизначення вимірювання vs оцінки)

## Наступні фази

### Фаза 12: Масовий імпорт пристроїв
**Ціль:** Масове додавання 50-1000 пристроїв одним CSV-файлом замість ручного призначення по одному.
**Тривалість:** 3-5 днів

- [ ] `POST /api/devices/import` — завантаження CSV, по кожному рядку: знайти очікуючий → призначити + MQTT-облікові дані
- [ ] `GET /api/devices/pending/export` — експорт списку очікуючих у CSV (заповнити в Excel → завантажити назад)
- [ ] `GET /api/devices/import/template` — порожній CSV-шаблон із заголовками
- [ ] Веб-інтерфейс: модалка імпорту — drag-and-drop, попередній перегляд, результат (призначено/пропущено/помилки)
- [ ] Веб-інтерфейс: кнопка "Імпорт CSV" на сторінці очікуючих пристроїв

**Результат:** Адмін скачує список очікуючих → заповнює метадані в Excel → завантажує назад → 500 пристроїв активовано за хвилину.

---

### Фаза 14: Бенчмаркінг флоту + виявлення аномалій
**Ціль:** Порівняння однотипного обладнання, автоматичне виявлення аномалій.
**Тривалість:** 2-3 тижні

- [ ] Сервіс базових показників флоту: середнє ± 2σ по моделі для робочого циклу, частоти аварій, відхилення температури
- [ ] Детектор аномалій: Z-score > 2.0 → подія аномалії + сповіщення
- [ ] REST API: бенчмарки, аномалії (по пристрою + по флоту)
- [ ] Сторінка аналітики флоту (нова): графік робочого циклу, таблиця викидів
- [ ] Деталі пристрою: вкладка Аномалії

**Результат:** "Холодильник №7 працює на 40% більше ніж середній Model X — перевірте ущільнення дверей."

---

### Фаза 15: Вебхуки + API-платформа
**Ціль:** Зовнішні інтеграції — CMMS, ERP, автоматизація.
**Тривалість:** 1.5-2 тижні

- [ ] Диспетчер вебхуків: події аварій/пристроїв/аномалій, підпис HMAC-SHA256, повтори + автовимкнення
- [ ] REST API: CRUD вебхуків, тестова доставка, журнал доставки
- [ ] API-ключі: автентифікація для міжсервісної взаємодії (альтернатива JWT)
- [ ] Специфікація OpenAPI 3.0 → `/api/docs`
- [ ] Веб-інтерфейс: сторінки вебхуків та API-ключів

**Результат:** Аварія → вебхук → CMMS створює наряд на ремонт. Автоматично.

---

### Фаза 16: Розширені звіти + PWA
**Ціль:** Друковані звіти для клієнтів, мобільний досвід.
**Тривалість:** 2-3 тижні

> Базовий HACCP експорт (CSV + PDF) вже реалізований у фазі 11b. Ця фаза розширює до запланованих звітів та доставки на email.

- [ ] Нові типи звітів: звіт про стан пристрою, огляд флоту, енергетичний звіт
- [ ] Заплановані звіти: щотижнева/щомісячна генерація → вкладення в email
- [ ] PWA: manifest.json, Service Worker, кешовані стани пристроїв офлайн, підказка встановлення

**Результат:** Клієнт отримує PDF-звіт щотижня на email. Технік встановлює PWA на телефон.

---

### Фаза 17: Рекомендації з обслуговування
**Ціль:** Автоматичні рекомендації з обслуговування на основі даних.
**Тривалість:** 1.5-2 тижні

- [ ] Движок правил: таймаути розморожування → "перевірте нагрівач", відхилення робочого циклу → "перевірте герметичність", мотогодини компресора → "планове ТО"
- [ ] REST API: рекомендації по пристрою, дія "відхилити"
- [ ] Push + вебхук-події для критичних рекомендацій
- [ ] Веб-інтерфейс: вкладка рекомендацій, бейдж на дашборді

**Результат:** Платформа каже: "Холодильник №3 — очистіть конденсатор, ефективність впала на 25%."

### Фаза 18: Самообслуговування орендарів + білінг
**Ціль:** SaaS бізнес-модель — клієнти реєструються і платять самостійно.
**Тривалість:** 4-6 тижнів

- [ ] Самореєстрація з підтвердженням email
- [ ] Обмеження плану: максимум пристроїв, максимум користувачів, функціональні обмеження
- [ ] Інтеграція білінгу (LiqPay / Stripe)
- [ ] Облік споживання: пристрої, обсяг телеметрії, API-виклики
- [ ] Веб-інтерфейс: реєстрація, білінг, порівняння планів, рахунки

| План | Пристрої | Користувачі | Можливості | Ціна |
|------|----------|-------------|------------|------|
| Free | 3 | 2 | Моніторинг, push | $0 |
| Pro | 50 | 10 | + Енергія, Health Score, Звіти, Вебхуки | ~$49/міс |
| Enterprise | ∞ | ∞ | + Аномалії, Рекомендації, API-ключі, SLA | Індивідуально |

**Результат:** Клієнт реєструється → додає пристрої → платить щомісяця.
